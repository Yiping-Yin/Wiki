# SwiGLU & Gated Activations

SwiGLU is a drop-in replacement for the activation function inside a transformer's feed-forward network (FFN). Instead of the classic `Linear â†’ ReLU â†’ Linear` sandwich, it uses a *gated linear unit* whose gate is a Swish (SiLU) nonlinearity. The gate multiplies one linear projection by a nonlinearly-activated sibling projection, letting the network learn input-dependent masking rather than applying a fixed elementwise nonlinearity.

Gated linear units were introduced by Dauphin et al. (2017) for convolutional language models, and Noam Shazeer repurposed them for transformer FFNs in "GLU Variants Improve Transformer" (2020, arXiv:2002.05202), empirically showing that SwiGLU and GeGLU outperform ReLU/GELU on T5-style pretraining at matched compute. Since then SwiGLU has become the default FFN activation in essentially every frontier open-weights LLM: PaLM, LLaMA 1/2/3, Mistral, Qwen, DeepSeek, and Gemma all ship it.

## Key idea

A standard transformer FFN is $\text{FFN}(x) = W_2 \,\sigma(W_1 x)$ with $\sigma \in \{\text{ReLU}, \text{GELU}\}$. A GLU-family FFN splits the up-projection into **two** matrices and multiplies their outputs elementwise, using one branch as a *gate*:

$$
\text{SwiGLU}(x, W, V, W_2) = \bigl(\,\text{Swish}_{\beta}(xW) \odot xV\,\bigr)\,W_2
$$

where $\text{Swish}_{\beta}(z) = z \cdot \sigma(\beta z)$ and $\sigma$ is the logistic sigmoid. In practice $\beta = 1$, which makes Swish identical to SiLU. GeGLU is the same recipe with GELU in place of Swish; ReGLU uses ReLU.

## How it works

The mechanism is a learned, data-dependent gate. For each hidden dimension $i$ in the FFN's intermediate space, the network computes two scalars from the input: a *value* $(xV)_i$ and a *gate* $\text{Swish}((xW)_i)$. Their product is the activation. Because the gate can smoothly range from near-zero (suppressing the channel) to values above one (amplifying it), the FFN learns per-token, per-channel routing instead of a fixed pointwise squash. This is the same intuition as LSTM input gates, but stateless and applied only inside the FFN.

A subtlety: SwiGLU has **three** weight matrices ($W$, $V$, $W_2$) where the baseline FFN has two ($W_1$, $W_2$). To keep parameter count (and FLOPs) matched when benchmarking, Shazeer shrinks the intermediate dimension by $2/3$. That is, if the baseline uses $d_{ff} = 4d$, the SwiGLU variant uses $d_{ff} = \tfrac{8}{3}d$, usually rounded to a multiple of 64 or 256 for hardware efficiency. LLaMA's $d_{ff} = 11008$ for $d = 4096$ comes from exactly this arithmetic: $\tfrac{8}{3} \cdot 4096 \approx 10923$, rounded up.

Gradients flow through both branches via the product rule: $\partial (g \odot v)/\partial x = g' \odot v + g \odot v'$. Unlike ReLU, neither branch kills gradient in a dead-zone, and the multiplicative interaction gives the optimizer a richer loss surface â€” empirically this shows up as lower pretraining loss at identical token/FLOP budgets, though nobody has a fully satisfying theoretical explanation. Shazeer himself famously closes the paper with "We offer no explanation as to why these architectures seem to work; we attribute their success, as all else, to divine benevolence."

## Code

```python
import torch
import torch.nn as nn
import torch.nn.functional as F

class SwiGLU(nn.Module):
    def __init__(self, dim: int, hidden: int | None = None):
        super().__init__()
        # shrink hidden by 2/3 to match param count of a ReLU FFN at 4*dim
        hidden = hidden or int(2 / 3 * 4 * dim)
        hidden = 64 * ((hidden + 63) // 64)  # round for hardware
        self.w_gate = nn.Linear(dim, hidden, bias=False)
        self.w_up   = nn.Linear(dim, hidden, bias=False)
        self.w_down = nn.Linear(hidden, dim, bias=False)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        return self.w_down(F.silu(self.w_gate(x)) * self.w_up(x))

if __name__ == "__main__":
    ffn = SwiGLU(dim=512)
    x = torch.randn(2, 16, 512)
    print(ffn(x).shape, "params:", sum(p.numel() for p in ffn.parameters()))
```

## Why it matters

The quality-per-FLOP improvement from SwiGLU is small in absolute terms (a few percent lower perplexity at matched compute) but it is *free*: no new hyperparameters, no training instability, trivial to implement, and it composes cleanly with every other transformer component. At frontier scale a few percent of compute is millions of dollars, so the switch was immediate and near-universal once LLaMA popularized it in 2023. Compared to alternatives: plain GELU is simpler and still fine for small models; GeGLU matches SwiGLU within noise; ReGLU is slightly worse but induces sparsity that some MoE-adjacent work exploits. Outside language, SwiGLU has quietly spread into vision transformers and diffusion backbones as well.

One practical consequence: any code that assumes the FFN has two matrices (weight surgery, LoRA targeting, quantization calibration, activation checkpointing heuristics) needs to be updated for three. LoRA configs for LLaMA-family models specifically list `gate_proj`, `up_proj`, `down_proj` for this reason.
