# FlashAttention v1 / v2 / v3

FlashAttention is an **IO-aware exact attention algorithm**: it computes the same softmax-attention output as the textbook formulation, but reorganizes the computation to minimize reads and writes to GPU high-bandwidth memory (HBM). Standard attention materializes the $N \times N$ score matrix $S = QK^\top$ and the probability matrix $P = \mathrm{softmax}(S)$ in HBM, which costs $O(N^2)$ memory and makes the kernel memory-bandwidth-bound. FlashAttention instead tiles $Q$, $K$, $V$, streams them through on-chip SRAM, and fuses the entire softmax-matmul pipeline into a single CUDA kernel.

Tri Dao introduced **v1** in May 2022 (Dao et al., "FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness"), **v2** in July 2023 with better work partitioning and parallelism across the sequence dimension, and **v3** in July 2024 with Hopper-specific optimizations (WGMMA, TMA, FP8, asynchrony between tensor cores and softmax). Today every major training and inference stack — PyTorch SDPA, vLLM, TGI, TRT-LLM, Megatron, xFormers — dispatches to FlashAttention as the default attention kernel for long-context transformers.

## Key idea

Naive attention reads/writes $O(N^2)$ bytes from HBM. The fix is to never materialize $S$ or $P$. FlashAttention uses the **online softmax** trick (Milakov & Gimelshein 2018): softmax is computed incrementally over tiles of keys by tracking a running max $m$ and running denominator $\ell$, rescaling the partial output on the fly.

$$
\mathrm{Attn}(Q,K,V) = \mathrm{softmax}\!\left(\frac{QK^\top}{\sqrt{d}}\right) V, \qquad \text{HBM reads: } O(N^2 d) \;\to\; O(N d + N d^2 / M)
$$

where $M$ is the SRAM size. Because the kernel is fused, memory traffic — not FLOPs — is what drops, and on A100 this yields a 2–4× wall-clock speedup while using $O(N)$ memory instead of $O(N^2)$.

## How it works

Split $Q$ into blocks $Q_i$ of size $B_r$ and $K,V$ into blocks $K_j, V_j$ of size $B_c$ chosen so that a tile fits in SRAM. For each $Q_i$, loop over all $K_j, V_j$, load them into SRAM, compute $S_{ij} = Q_i K_j^\top / \sqrt{d}$, and update the output $O_i$ along with per-row statistics $(m_i, \ell_i)$ using the online softmax recurrence:

$$
\begin{aligned}
m_i^{\text{new}} &= \max(m_i,\; \mathrm{rowmax}(S_{ij})) \\
\ell_i^{\text{new}} &= e^{m_i - m_i^{\text{new}}}\ell_i + \mathrm{rowsum}(e^{S_{ij} - m_i^{\text{new}}}) \\
O_i^{\text{new}} &= \mathrm{diag}(e^{m_i - m_i^{\text{new}}})\,O_i + e^{S_{ij} - m_i^{\text{new}}} V_j
\end{aligned}
$$

After the inner loop, divide by $\ell_i$ once. The result is numerically identical to standard softmax attention — this is *exact*, not an approximation. The backward pass recomputes $S$ and $P$ tile-by-tile from saved $(m,\ell)$ instead of storing them, trading a modest FLOP increase for a massive memory-bandwidth win.

**v2** flips the loop order: the outer loop runs over $Q$ blocks and is parallelized across SMs (not just batch × heads), so short-batch long-sequence workloads saturate the GPU. It also reduces non-matmul ops (which run on slow CUDA cores, not tensor cores), pushing A100 utilization from ~25% to ~70% of peak. **v3** targets H100: it overlaps GEMMs with softmax via warp-specialized producer/consumer pipelines using TMA for async copies and WGMMA for async tensor-core ops, and adds an FP8 variant that reaches ~1.2 PFLOPs/s — about 75% of H100 peak.

## Code

```python
import torch

def flash_attention_forward(Q, K, V, Br=64, Bc=64):
    N, d = Q.shape
    O = torch.zeros_like(Q)
    m = torch.full((N,), -float('inf'), device=Q.device)
    l = torch.zeros((N,), device=Q.device)
    scale = d ** -0.5
    for i in range(0, N, Br):
        Qi = Q[i:i+Br]
        Oi = torch.zeros_like(Qi)
        mi = torch.full((Qi.shape[0],), -float('inf'), device=Q.device)
        li = torch.zeros((Qi.shape[0],), device=Q.device)
        for j in range(0, N, Bc):
            Kj, Vj = K[j:j+Bc], V[j:j+Bc]
            Sij = (Qi @ Kj.T) * scale
            m_new = torch.maximum(mi, Sij.max(dim=-1).values)
            Pij = torch.exp(Sij - m_new[:, None])
            alpha = torch.exp(mi - m_new)
            li = alpha * li + Pij.sum(dim=-1)
            Oi = alpha[:, None] * Oi + Pij @ Vj
            mi = m_new
        O[i:i+Br] = Oi / li[:, None]
        m[i:i+Br], l[i:i+Br] = mi, li
    return O
```

## Why it matters

Before FlashAttention, training with sequences beyond ~2k tokens meant either gradient checkpointing the attention block (slow) or switching to an *approximate* sparse/low-rank attention (lossy, and almost always worse in quality). FlashAttention removed that tradeoff: you get exact attention at $O(N)$ memory, which is what unlocked the 8k → 32k → 128k → 1M context race. GPT-4-class training runs, Llama-2/3, Mistral, and essentially every open model since 2023 use it by default. It is also the substrate for Paged/Ring/Striped attention variants used in vLLM and long-context inference.

Compared to alternatives: Linformer/Performer/Longformer are approximate and degrade quality; xFormers' memory-efficient attention (Rabe & Staats 2021) achieves $O(N)$ memory but is not IO-aware and is ~2× slower. FlashAttention's edge is that it is *both* exact *and* hardware-optimal.

<Callout type="tip">
The speedup does **not** come from fewer FLOPs — FlashAttention actually does *more* FLOPs than naive attention (it recomputes $S$ in the backward pass). The win is purely from cutting HBM traffic. This is the canonical lesson of modern GPU kernels: on tensor-core hardware, memory bandwidth is the bottleneck, not arithmetic, and "extra compute to save a memory round-trip" is almost always the right trade.
</Callout>
